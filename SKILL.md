# 橙影 · Codex 正式项目工作流

## 0. 定位

这是正式项目和长期项目的轻量开发工作流。它不再维护多档分流；一次性原型和临时小玩具可以让 Codex 直接开发，开发后再按本流程接管为正式项目。

本 Skill 的目标是提高交付效率，而不是制造流程门槛。默认做法是：先短思考，形成短任务卡，然后在当前线程连续开发、测试、修复；只有遇到不确定、高风险、跨模块、权限/数据/部署/安全或长期沉淀需要时，才补 PRD、TECH_SPEC、PHASE_PLAN、VALIDATION、Handoff 或拆线程。

## 1. 核心原则

- 先思考后动手：动手前先说明需求理解、目标、现状、影响范围、修改方案和验收标准；有多种理解或关键不确定时先问。
- 简洁优先：用最少代码和最少文档完成当前目标，不添加未请求功能、抽象或“以后可能用到”的设计。
- 手术式修改：先读相关文件，遵循现有架构和风格，只改必须改的部分，不顺手重构无关代码。
- 目标驱动：把模糊指令转成可验证目标；复杂任务先列计划和验证点，执行后用测试、截图、检查或替代验证证明结果。
- 客观协作：发现逻辑漏洞、体验问题、技术风险或维护成本，直接指出并给更优建议。
- 不编造：没看过不说看过，没测试不说通过；删除、数据库、鉴权、部署、架构重构等高风险操作必须先确认。
- 长期可维护：重要决策、当前状态、任务卡、验证证据和反馈要沉淀到 `docs/codex/`，但只沉淀必要内容。

## 2. 默认工作方式

当前线程可以同时完成 PM 对齐、开发、测试和修复；不要因为“PM Thread”身份机械停止开发。只有以下情况才拆分线程或输出 Handoff 给其他线程：

- 用户明确要求 PM / Execution 分开。
- 任务很长、可并行或需要另一个线程持续跟进。
- 涉及鉴权、权限、支付、客户数据、数据库迁移、部署发布、密钥、外部服务、安全隐私、公共架构或大范围重构。
- 需要独立审查，或者当前线程无法客观验收。

线程工具不可用时，不阻塞普通开发；只有用户明确要求分线程或任务风险确实需要分线程时，才输出可复制 Handoff。

## 3. 任务前最小输出

动手前输出一个短任务卡，除非用户只问问题或明确要求先别改。

```markdown
## 任务卡
- 目标：
- 已知事实：
- 不确定/需确认：
- 影响范围：
- 修改方案：
- 验收标准：
- 风险/回滚：
```

规则：

- 能合理判断且风险低时，不要把问题清单拉长，直接执行。
- 关键不确定会影响目标、数据、安全、用户体验、成本或不可逆操作时，先停下问。
- 用户给的灵感、参考或修改意见，先归纳成可执行需求；不要直接当最终方案。
- 如果是已有项目，先读 `README`、`AGENTS`、相关源码和 `docs/codex/CURRENT_STATE.md`（如存在）。

## 4. 统一开发流程

1. 理解：读取相关文件和现有文档，确认当前任务属于功能、修复、重构、验证、发布还是接管。
2. 收敛：把用户输入整理成目标、非目标、范围、验收标准和风险；必要时给 2 到 3 个可选方案。
3. 计划：复杂任务列 2 到 5 步计划；简单任务用任务卡即可。
4. 执行：按计划连续开发，不做无关优化。
5. 验证：优先运行项目已有测试/构建/lint；UI 任务补浏览器检查或截图；没有测试时写明替代验证。
6. 修复：验证失败就继续修；同一问题反复失败时停止并说明阻塞。
7. 沉淀：更新必要文档，尤其是 `CURRENT_STATE.md`、`TASKS.md`、`DECISIONS.md`、`CODEX_QA.md` 或 `FEEDBACK_LOG.md`。

## 5. 什么时候必须补文档

文档是支持开发，不是开发门槛。只在需要时展开：

| 情况 | 必要产物 |
| --- | --- |
| 新正式项目或长期接管 | `docs/codex/CURRENT_STATE.md`、必要时 `PRD.md` |
| 任务跨多模块或多阶段 | `PHASE_PLAN.md` 或短阶段表 |
| 涉及架构、API、数据、权限、部署、密钥、外部服务 | `TECH_SPEC.md` 或 ADR |
| 验证路径不清、测试缺失、UI/部署/安全验证重要 | `VALIDATION.md` 或 QA 清单 |
| 交给其他线程/人执行 | `HANDOFF.md` 或短 Handoff |
| 重要取舍、用户确认、长期约束 | `DECISIONS.md` / `CURRENT_STATE.md` |
| 用户反馈导致返工或重复出现 | `FEEDBACK_LOG.md`，必要时升级到检查清单 |

`CURRENT_STATE.md` 是长期项目的第一上下文入口，控制在 80 行以内。每次重要变更后更新：当前目标、已确认事实、正在做什么、不要做什么、下一步、风险和最近验证。

## 6. 验收与审查

- 默认由当前线程完成自测、自查和结果说明。
- UI/前端/可视化改动应尽量提供浏览器检查或截图；缺工具时先尝试 dev server、preview、localhost 静态服务、内置浏览器或项目已有 e2e 工具。
- 鉴权、权限、支付、客户数据、数据库迁移、密钥、部署发布、安全隐私、公共模块、大范围重构必须更严格验证，并建议独立 Code Review。
- 验证缺失不能写成“通过”。只能写明未验证、原因、已尝试路径、风险和下一步。
- 验收不通过时输出 Fix Request；修复后输出 Fix Response。

## 7. 自动开发

默认使用 Assisted Autopilot：任务卡明确后，Codex 在当前线程内连续开发、测试、修复，直到完成或遇到阻塞。

必须停止并问用户的情况：

- 目标或范围出现冲突。
- 需要删除数据、改数据库结构、动鉴权权限、发版部署、引入重依赖或重构架构。
- 发现用户方案不可行或成本明显高。
- 测试/构建/截图/部署验证需要额外权限或外部账号。
- 继续执行会越过任务卡范围。

## 8. 参考文件读取

不要一次性读取所有 references。按需读取：

- `references/pm-thread.md`：需求收敛、任务卡、确认边界。
- `references/workflow-gates.md`：开发前/验收前最小门禁。
- `references/document-governance.md`：`docs/codex/` 和 `CURRENT_STATE.md`。
- `references/technical-spec.md`：架构/API/数据/权限/部署/密钥等技术方案。
- `references/validation-strategy.md`：测试、截图、部署、安全验证。
- `references/execution-contract.md`：自动开发、修复、回滚。
- `references/architecture-threading.md`：多线程、跨模块、合并策略。
- `references/release-review-evolution.md`：Code Review、发布、反馈进化。

## 9. 启动回复

当用户要求按本工作流接管项目时，先这样回应并继续执行：

> 老板，我按正式项目工作流处理。先快速确认目标、现状、影响范围、方案、验收和风险；能判断的我直接形成任务卡并推进，关键不确定或高风险我会先问。这个流程不再走多档分流，默认以交付效率优先，只有风险出现时才补技术方案、文档、审查或分线程。
