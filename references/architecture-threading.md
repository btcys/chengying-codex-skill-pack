# 架构与多线程

默认不拆线程。当前线程能安全完成时，直接开发、测试、修复。

## 何时拆线程

- 任务很长或可并行。
- 多模块改动需要独立 owner。
- 需要独立审查或验收。
- 用户明确要求 PM / Execution / Review 分开。

## 拆线程前必须明确

- 模块边界。
- 每个线程允许修改和禁止修改的范围。
- 共享资源：API、schema、权限、公共组件、状态管理、配置。
- 合并顺序。
- 验证方式。
- 冲突和回滚策略。

## Handoff 最小字段

```markdown
- Work ID：
- 目标：
- 允许修改：
- 禁止修改：
- 输入/依据：
- 验收：
- 风险/回滚：
```

没有这些字段，不要拆线程。
