## 目标

架构梳理用于回答一个问题：这里改完，会不会让别的地方出问题？

多线程分工前必须先做架构梳理，不能只按功能名称拆线程。

技术方案和架构影响面分工：

- `TECH_SPEC.md` 回答怎么实现、为什么这样实现、技术取舍和回滚方案。
- 架构影响面回答会影响哪些模块、接口、数据、权限、构建和测试。
- Lean 可以合并写在 Handoff；Standard/Enterprise 涉及核心契约时必须分清这两类信息。

## 多线程判断时机

多线程不是到 Goal 执行时临时决定，也不是看到项目大就立即开启。真正“开几个线程开发”放在 Goal Prompt / Execution 前一步，按三段判断：

1. PM 启动 / 第一轮：只做多线程预判，输出 `single-thread likely` / `multi-thread possible` / `defer` 和理由；不得创建执行线程。
2. 技术方案 / 架构影响面阶段：只做可拆性分析和候选拆分。确认模块边界、数据/API 契约、共享资源、风险模块和潜在冲突点，但不在这里最终决定线程数量。
3. 阶段计划与验证方案完成后，Goal Prompt / Execution 前：输出执行前线程方案，正式确定 `single-thread` / `N threads` / `defer`。此时必须写清 Work ID 字段、每个线程的目标、允许修改范围、禁止修改范围、上下游依赖、验收方式、合并顺序和冲突处理方式；用户授权后再登记 Work ID 并启动并行开发。

如果执行中发现某个线程改变了其他线程的输入输出，或共享文件/公共契约发生冲突，立即暂停并回到 PM Thread 或技术方案阶段重新判断。

以下情况不得开启多线程：

- 需求、产品原型、技术方案、验收标准或验证方式仍不清楚。
- 模块边界、数据/API 契约、权限模型或共享资源 owner 未明确。
- 多个线程需要同时修改同一公共组件、schema、API client、状态管理或构建配置。
- 没有合并顺序、回滚策略、冲突处理 owner 或跨模块验证矩阵。
- Lean 小任务、局部 bugfix、单页 Demo 或失败代价低的短任务。

## 架构影响面必填项

每个开发任务至少确认：

- 模块清单：每个模块的职责、输入、输出、owner。
- 依赖关系：哪些模块依赖它，它又依赖哪些模块。
- 共享资源：公共组件、hooks、store、API client、schema、配置、权限、上传、缓存、队列。
- 影响面表：本次任务可能影响的页面、接口、数据结构、权限、构建、测试。
- 线程责任矩阵：每个线程允许修改范围、禁止修改范围、需要同步的上下游。
- 冲突点：同一文件、同一公共模块、同一数据模型是否被多个线程同时触碰。
- 合并顺序：先合基础设施、数据模型、公共契约，再合业务功能和 UI。
- 验证矩阵：每个线程完成后必须验证哪些跨模块路径。

## 影响面检查问题

在改代码前检查：

- 这里改完，其他页面是否依赖同一个组件、hook 或 store？
- 这里改完，接口返回结构是否影响其他调用方？
- 这里改完，数据库 schema 是否需要迁移和兼容？
- 这里改完，权限和登录状态是否影响其他入口？
- 这里改完，构建配置、环境变量、路由、中间件是否影响全局？
- 这里改完，测试覆盖是否只验证了局部而没有验证主链路？

## 多线程适用条件

同时满足时，才建议多线程：

1. README / AGENTS / PROJECT_BRIEF 已建立。
2. MVP 已冻结。
3. 模块边界清晰。
4. 数据模型初步稳定。
5. 公共组件边界清晰。
6. Git 分支策略明确。
7. 每个线程能限制修改范围。
8. 有合并顺序和回滚策略。
9. 架构梳理已完成，并明确模块依赖、共享资源、上下游影响面。
10. 线程责任矩阵已确认，每个线程都有唯一 owner、允许修改范围和禁止修改范围。

## 线程类型

- PM Thread：需求梳理、范围确认、任务拆解、交接，不写代码。
- Long-lived Core Thread：核心架构、状态、领域模型、数据模型，服务长期主线。
- Long-lived Feature Thread：稳定业务模块，适合持续迭代。
- Phase Acceptance Thread：每轮/每阶段计划完成验收，不写业务代码，不合格打回 Execution Thread。
- Code Review Thread：独立审查正确性、安全、影响面、测试缺口，不直接做业务开发。
- Release Thread：隐私审计、打包发布、版本文档和发布后验证。
- Temporary Hotfix Thread：紧急修复，完成后尽快合并归档。
- Temporary Experiment Thread：实验性 UI 或功能，验证后合并或放弃，并归档记录结论。
- Temporary Spike Thread：技术调研、开源验证、风险验证，默认不长期保留。

## 长期线程与临时线程

长期线程优先服务长期项目主线：

- 核心架构
- 稳定业务模块
- 数据模型和 API 契约
- 持续迭代的用户路径

临时线程只用于：

- 紧急修复
- 一次性实验
- 技术风险验证
- 强边界的小任务

临时线程完成后必须立即进入：

1. 合并
2. 放弃
3. 归档

并更新 `THREADS.md` / `MERGE_PLAN.md`。

## 线程拆分规则

- 能独立交付、独立测试、独立回滚的任务，才适合拆成独立线程。
- 共享模块没有 owner 时，不允许多个线程并行修改。
- API 契约、数据模型、权限模型、状态管理先由架构线程或核心线程稳定后，功能线程才能并行。
- 如果一个线程的修改会改变其他线程的输入输出，必须先暂停并回到 PM Thread 重新确认。
- 临时线程只能在强边界内工作，完成后必须把结论写入 `THREADS.md` / `MERGE_PLAN.md`。

## 线程责任矩阵模板

```markdown
| Thread | 类型 | Owner | 目标 | 允许修改 | 禁止修改 | 上游依赖 | 下游影响 | 验收 | 回滚 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| PM | PM | Codex | 需求与交接 | docs/codex/* | 源码 | 用户输入 | Execution | 交接包确认 | 不进入执行 |
| core-* | Long-lived Core | Codex | 核心契约 | schema/api/shared | feature UI | PM Handoff | Feature | typecheck/build | revert branch |
```

## 合线前检查

合线前必须：

- typecheck 通过
- lint 通过
- build 通过
- 不修改无关模块
- 对照架构影响面确认没有破坏其他模块
- 更新 CHANGELOG.md
- 更新 TASKS.md
- 涉及架构时更新 DECISIONS.md
- 涉及多线程时更新 THREADS.md / MERGE_PLAN.md
- 临时线程标记“已合并 / 已放弃 / 已归档”
- 输出风险说明
- 可回滚

## 架构可视化

按项目大小执行，不要过度设计：

- Level 1：只输出模块列表。
- Level 2：输出简单依赖关系图，可用 Mermaid。
- Level 3：输出模块结构图、依赖关系图、数据流图、多线程结构图、风险模块标记。

架构图必须服务执行和验收，不要为画图而画图。
