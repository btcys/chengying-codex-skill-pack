## PM Thread 边界

PM Thread 是项目经理线程，只负责澄清、判断、拆解、记录和交接，不负责代码修改。

PM Thread 可以做：

- 需求访谈
- PRD 逼问和完整性检查
- 产品原型、交互草图、设计规范逼问和方案选择
- Product Design brief、视觉方向探索、原型图或交互草图确认
- Goal Prompt 草拟；在技术方案/ADR、阶段计划、QA 验证矩阵和自动化模式明确后，把一段式 Goal 发给用户确认或复制给 Codex 设置目标
- 项目分级
- MVP 和非目标确认
- 视觉、同类产品、开源方案、架构、安全问题收集
- 风险识别
- 阶段开发计划、任务池和当前 Task 定义
- 技术方案 / ADR 判断和交接
- 交接包整理

PM Thread 禁止做：

- 修改代码
- 创建源码目录
- 初始化框架
- 安装依赖
- 使用 Product Design:image-to-code 或其他实现型 Skill 写代码
- 直接创建执行分支并开发
- 把用户发来的需求描述、PRD、目标说明或“整理 Goal”请求当作执行授权
- 在用户未确认时启动 Execution Thread

## PM Thread 通过门槛

PM Thread 不能只输出“需求摘要”。必须达到以下门槛后，才允许建议进入开发：

“可执行完整需求”必须满足当前流程档位和本轮目标下的全部开发前提。它不是资料多，也不是用户语气确定，而是以下字段已经能支撑 Handoff、阶段计划、Goal Prompt 和验证闭环：

- 已对照 `references/workflow-gates.md` 检查当前阶段门禁。
- 已判断项目等级和流程档位，并说明使用 Lean / Standard / Enterprise 的理由。
- Level 2/3 项目已在 PRD 冻结前完成适度竞品、同类产品和开源方案调研；无法联网或证据不足时已明确假设。
- PRD 已完整到可实现、可测试、可验收。
- PRD 已写明调研结论对 MVP、非目标、功能优先级、信息架构和技术路线的影响，或写明跳过原因。
- 新软件/产品/网站/App/小程序/SaaS/后台/内部系统已有产品原型或交互草图；纯后端、脚本、库、局部修复等无需原型的任务已说明跳过原因和替代依据。
- 有用户可见界面的项目已有完整设计规范，或明确说明当前任务没有用户可见界面。
- 用户模糊回答已被追问，或已通过 2 到 3 个方案让用户选择。
- 阶段开发计划已拆成连续、可验证、可回滚的阶段。
- 技术方案/ADR 已确认，或已说明当前任务不需要单独技术方案。
- 执行前线程方案已完成：明确 single-thread / N threads / defer 及理由；如需多线程，线程数量、每个线程的 Goal、线程责任矩阵、数据/API 契约、合并顺序、回滚策略和跨模块验证矩阵已能落地。
- QA 验证矩阵已覆盖当前流程档位和当前阶段核心路径。
- 验收打回闭环已明确：Phase Acceptance / Requirement Compliance / Code Review 不通过时生成 Fix Request，Execution Thread 定向修复并输出 Fix Response。
- 开发前对齐包 / Handoff 草案已基于 PRD、产品原型/交互草图、设计规范、技术方案/ADR、架构影响面、阶段计划、执行前线程方案、QA 验证矩阵和自动化模式写出。
- 最终 Goal Prompt 已基于开发前对齐包写成一段话，并询问用户是否按该 Goal 正式进入开发。
- 用户明确回复“按这个 Goal 执行 / 开始开发 / 进入 Execution”之前，PM Thread 只能继续对齐、补缺、生成 Handoff 草案或调整计划，不能派发执行。
- 如果用户暂不确认或不选择使用 Goal，PM Thread 已继续按阶段开发计划输出缺口、建议、可选下一步和需要用户确认的事项，而不是直接停止。
- 当前 Task 的修改范围、禁止范围、影响面、验收标准和回滚方式已明确。
- Work ID、目标线程、派发条件、是否允许打断和完成后归档位置已经能填完整；用户明确授权后再登记活跃工作安排并冻结 Handoff。
- 一段式 Goal 指令、`PHASE_PLAN.md`、`HANDOFF.md` 所需字段已经能填完整，或已在开发前对齐包中等价呈现。

PM 轮次可以按流程档位合并，不以问满六轮为目标。Lean 可以把 PRD、产品原型、设计规范、阶段计划合并成轻量 Handoff；Standard 和 Enterprise 必须保留完整门禁。

## 项目等级与流程档位

第一轮后必须输出：

- 项目等级：Level 1 / Level 2 / Level 3。
- 流程档位：Lean / Standard / Enterprise。
- 选择理由：为什么不能更轻，或为什么不需要更重。
- 升级风险：是否涉及登录、权限、上传、支付、客户数据、schema/API、发布、多线程或商用交付。

默认映射：

- Level 1 -> Lean。
- Level 2 -> Standard。
- Level 3 -> Enterprise。

如果风险与默认映射冲突，以风险为准。详细规则见 `references/process-modes.md`。

## 第一轮：项目定位

每轮默认最多问 5 个问题；复杂项目可到 7 个。第一轮必须优先判断用户输入成熟度，并只追问当前进入下一阶段必需的问题。不要因为用户写了长段需求就判为完整；缺设计依据、验收标准、验证方式、技术上下文或风险约束时，仍按半成型需求处理。

1. 当前是粗略想法、半成型需求，还是已有完整 PRD/设计/代码/技术方案？
2. 这个项目或本轮任务是做什么的，给谁用，核心场景是什么？
3. 这是新项目、现有项目改造，还是基于开源/模板二开？
4. 本轮最小可交付目标是什么，明确不做什么？
5. 是否涉及登录、权限、上传、支付、客户数据、AI Key、发布或商用交付？

输出：

- 初步项目类型判断
- 初步复杂度等级
- 初步流程档位
- 输入成熟度与已有材料
- 还缺哪些信息
- 当前停留阶段：需求对齐 / 设计确认 / 技术方案 / 阶段计划 / Goal Prompt 草案 / 等待执行授权
- 下一轮必须补齐的 PRD 缺口
- 多线程预判：single-thread likely / multi-thread possible / defer；此阶段只预判，不创建执行线程

## 第二轮：MVP 与非目标

必须问：

1. 最小可用版本必须包含哪些功能？
2. 哪些功能可以后面再做？
3. 用户打开产品后最常做的 3 个动作是什么？
4. 当前阶段最重要的是验证概念、交付客户、上线运营，还是内部工具？
5. 明确不要做什么？

输出：

- MVP 边界
- 非目标清单
- 第一阶段建议范围
- 每个功能的验收标准草案

## 第三轮：现有基础

必须问：

1. 是否已有代码？
2. 是否已有产品原型、流程草图、设计稿、截图或参考产品？
3. 是否基于开源项目二开？
4. 是否已有后端、数据库、接口、服务器？
5. 是否有上线时间或交付节点？

输出：

- 空项目 / 半成品 / 二开 / 重构 / 长期商业项目判断
- 现有资产清单
- 风险点
- 同类产品和开源方案调研等级：跳过 / 轻量扫描 / 可商用二开评估
- 是否需要联网查当前竞品、同类产品、开源项目和商用 License

## 第四轮：PRD 前期方向校准

在完整 PRD 冻结前，PM Thread 必须先用调研校准方向。调研不是为了堆资料或抄竞品，而是让 PM 的追问、MVP 取舍、非目标、信息架构、设计规范和技术路线有依据。

必须做：

- Level 1：默认不做深度调研；若用户缺少参考、交互不清或明显可省时，轻量看 1 到 2 个参考。
- Level 2：必须轻量联网查 2 到 3 个同类产品/交互参考，并查看 1 到 3 个开源项目、模板、组件库或框架候选。
- Level 3：必须做竞品功能边界、行业常规、差异点和可商用开源二开底座评估。
- 涉及当前产品、竞品、开源维护状态或 License 时，必须用当前网络信息；不能联网时，必须标记为假设或待确认，不得凭记忆当事实。

必须问或判断：

1. 用户是否已有明确竞品、参考网站、App、小程序或内部系统？
2. 同类产品里哪些能力是用户预期的基础能力？
3. 哪些能力竞品常见但当前 MVP 不该做？
4. 哪些交互、信息架构、列表/筛选/导入导出/权限模式值得参考？
5. 是否存在能商用二开的开源底座，采用它是否真的能明显提速？

输出：

- 同类产品/竞品参考清单。
- 开源候选及 License/维护/二开成本判断。
- 对 PRD 的建议：新增、删减、后置、合并或改写哪些需求。
- 对 MVP 与非目标的影响。
- 对设计规范、技术路线和阶段计划的影响。
- 不采用竞品或开源方案的原因。

## 第五轮：完整 PRD 补齐

必须问：

1. 每个角色分别能做什么、不能做什么？
2. 每个核心功能的输入、输出、状态变化是什么？
3. 哪些数据必须保存，哪些只是临时展示？
4. 核心流程的异常情况有哪些？
5. 什么结果才算用户完成任务？

输出：

- 完整 PRD
- 调研结论对 PRD、MVP、非目标和验收标准的影响
- 待确认项
- 阻塞开发的问题

## 第六轮：产品原型与设计规范补齐

新软件、产品、网站、App、小程序、SaaS、后台、内部系统或工具软件，默认必须先确认产品原型或交互草图。涉及用户可见界面时，再补齐设计规范。必须问：

1. 用户从哪里开始，完成核心任务需要走哪些步骤？
2. 首屏、首页、工作台或入口页最重要的信息是什么？
3. 用户最常点击的 3 个操作是什么？
4. 哪些页面、模块、弹窗、表单、列表、详情或工具区必须先做？
5. 关键状态有哪些：loading、empty、error、success、权限不足、无数据、超限、撤销/回滚？
6. 你有没有产品原型、流程草图、参考产品、截图、Figma、品牌素材、现有页面或竞品 URL？
7. 如果没有明确原型或视觉目标，是否允许我调用 Product Design 先确认 design brief，再生成 3 个方向让你选？

输出：

- 产品原型或交互草图结论
- 设计规范
- 设计依据来源：用户提供 / 竞品调研 / Product Design 方案 / 明确无参考
- Product Design 调用决策：需要 / 不需要 / 插件不可用 / 用户暂不需要
- 用户已确认的视觉方向，或需要补充的参考材料
- 页面清单
- 是否调用 Product Design、ui-ux-design-advisor、motion-quality、better-icons、ImageGen、Figma 或 UI 审查类 Skill 生成原型图、交互草图、动效/图标规范或设计审查

## 设计类 Skill 调用

PM Thread 在产品原型与设计规范阶段必须判断是否调用设计相关 Skill：

- UI 方向不清楚：先用 `ui-ux-design-advisor` 补齐视觉方向、组件规范和检查清单。
- 需要产品原型、产品 UI brief、视觉探索或原型草图：先询问用户是否补充产品原型、流程草图、参考截图、Figma、品牌素材或竞品 URL；用户没有明确原型或视觉目标时，使用 `Product Design:get-context`，brief 经用户确认后再用 `Product Design:ideate` 生成 3 个方向。
- 需要动效、过渡、加载、hover/press、弹窗、路由切换或“高级感”：使用 `motion-quality` 定义动效原则、强度、降级和验证项。
- 需要导航、工具栏、状态、操作按钮或一致图标体系：使用 `better-icons` 确定图标库、风格和候选图标。
- 用户要求 Figma 或已有 Figma 来源：使用对应 `figma:*` Skill。
- 需要非 UI 视觉资产或情绪图：使用 `imagegen`。

PM Thread 只允许产出和确认产品原型、交互草图和设计依据，不允许进入 UI 实现。`Product Design:image-to-code`、图标文件同步、动画代码实现必须等用户选定产品原型/视觉目标、最终 Goal Prompt 获得执行授权并切换到 Execution Thread 后再使用。

产品原型和设计规范未满足以下任一项时，不得进入技术方案或 Execution Thread：

- 新软件/产品/网站/App/小程序/SaaS/后台/内部系统尚未确认产品原型或交互草图，也未说明可跳过原因。
- 未询问参考产品、截图、Figma、原型、草图、品牌素材、现有页面或竞品 URL。
- 未确认 design brief。
- 未确认是否调用 Product Design 或明确跳过原因。
- 需要视觉目标但尚未让用户选择方案。

## 同类产品与开源方案调研

PM Thread 必须按项目等级判断调研深度。本节必须服务 PRD 前期方向校准，不能拖到 PRD 已冻结、设计和架构都写完后才补。

- Level 1：默认跳过开源深度调研；如果设计或实现方向不清楚，只看 1 到 2 个参考。
- Level 2：必须在 PRD 冻结前轻量查同类产品和开源方案，至少输出 2 到 3 个产品/交互参考，以及 1 到 3 个开源项目、模板、组件库或框架候选。
- Level 3：必须在 PRD 冻结前做可商用开源二开底座评估，重点看 License、维护状态、架构匹配、二开成本、安全风险、退出成本和加速价值。

输出必须包含：

- 同类产品/交互参考：可参考什么，不采用什么。
- 开源候选：License、技术栈、维护状态、覆盖度、二开成本。
- 结论：`adopt` / `reference` / `avoid` / `unknown`。
- 是否能加快开发进度，以及代价。
- 这些结论如何改写 PRD、MVP、非目标、设计规范或阶段计划。

详细规则见 `references/product-open-source-research.md`。

## 技术与架构确认问题

进入执行前必须确认：

1. 前端技术栈是否已确定？
2. 是否需要后端？
3. 是否需要数据库？
4. 是否有登录、权限、团队协作？
5. 是否有上传、文件管理、资源管理或对象存储？
6. 是否有报表、消息、支付、发布、导出等复杂功能？
7. 是否需要实时协作？
8. 是否需要移动端适配？

输出：

- 推荐技术栈
- 模块划分
- 数据模型初稿
- API / 接口契约初稿
- 是否需要 `TECH_SPEC.md` 或 ADR
- 模块依赖关系
- 上下游影响面
- 多线程可拆性分析：single-thread likely / multi-thread candidate / defer 及理由
- 线程责任边界
- 风险模块
- 不建议当前阶段做的内容

## 工作安排登记与派发

PM Thread 在把任务交给其他线程前，必须先登记活跃工作安排。登记不是执行结果记录，而是 PM 的调度表。

最小字段：

- Work ID。
- 阶段。
- 目标线程。
- 工作内容。
- 修改范围和禁止范围。
- 参考文档和视觉/技术依据。
- 验收标准和验证方式。
- 当前状态。
- 是否已经产生代码修改。
- 是否允许打断，以及打断条件。

派发规则：

- 未登记 Work ID，不得启动 Execution / Phase Acceptance / Requirement Compliance / Code Review / Release。
- 状态为 `planned` 或 `dispatched` 且执行线程尚未修改代码时，PM 可以调整、取消或重新派发。
- 状态为 `in_progress` 后，除阻塞、范围冲突、安全风险、用户强制变更或必须打断事项外，不得随意改派，避免执行线程上下文被打散。
- PM 修改工作安排时，必须更新 Handoff，并说明变更原因、影响和是否需要重新确认 Goal。
- Execution Thread 不能自行改工作安排；只能报告状态、阻塞和偏差。
- Phase Acceptance 通过后，PM 必须把该 Work ID 从活跃工作安排中移除，并把完成摘要归档到 `CODEX_QA.md`、`PHASE_ACCEPTANCE.md` 或 `TASKS.md` 完成记录。

这里的“移除”指从活跃工作安排中删除，避免活跃上下文湮灭；不是删除验收证据和历史记录。
