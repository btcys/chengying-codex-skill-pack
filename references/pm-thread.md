# Goal Contract 与 Product Spec Builder

## PM 主控目标

PM 主控线程不是人工流程管理员，而是 AI 编排器：把用户目标自动转成 PRD、设计、开发计划和子线程任务。

## Goal Contract

```markdown
## Goal Contract
- 目标：
- 完成标准：
- 验收方式：
- 非目标：
- 关键不确定：
```

只有关键不确定影响方向、范围、数据、安全、成本或验收时才问用户。

## Product Spec Builder

AI 自动生成短 PRD：

```markdown
# PRD

## 目标用户
## 核心场景
## MVP
## 非目标
## 功能边界
## 验收标准
## 待确认
```

用户给的想法、参考资料和灵感必须先归纳成可执行需求，不得直接当最终方案。

## 复杂项目调研门槛

复杂项目在 PRD 冻结前先做时间盒调研，默认输出 `docs/codex/PRODUCT_RESEARCH.md`，再写 PRD：

- 触发条件：业务域不熟、竞品形态明显、功能链路长、多角色/多端/后台、外部 API/数据源多、技术实现不确定、可能存在成熟开源底座。
- 竞品/参考产品：找 3-5 个同类产品或页面，记录核心路径、信息结构、关键功能、定价/权限/边界、可借鉴点和不照搬点。
- 开源候选：找 3-5 个可二开的项目，记录 License、技术栈、维护状态、功能覆盖度、二开成本、安全/商用风险和 adopt/reference/avoid 结论。
- 没有联网、用户禁止联网或资料不足时，明确写“未完成调研”和原因，不得假装已调研。
- 调研结论只作为 PRD 对齐依据，不得把竞品或开源项目直接当最终方案；采用开源底座、换技术栈或引入重依赖必须先确认。

## PRD 可开发门槛

PRD 不是愿望清单，必须逼问到可实现、可测试、可验收：

- 复杂项目 PRD 必须引用 `docs/codex/PRODUCT_RESEARCH.md` 的结论或说明为什么不需要调研。
- 每个核心功能写清输入、输出、数据来源、权限/角色、状态/错误态、验收方式。
- 每个用户路径写清起点、动作、系统反馈、完成态和失败态。
- 每个范围写清 MVP、非目标和暂不做。
- 出现“好看、智能、方便、自动、完善、类似某产品”时，必须继续追问或翻译成具体要求。
- 关键问题影响开发实现时，PRD 不得标记为 ready。

## 人工介入点

- Goal 方向不对。
- MVP / 非目标需要取舍。
- 参考资料用途不清。
- 验收方式不清。
