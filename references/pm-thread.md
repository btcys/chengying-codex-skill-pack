# 需求收敛与任务卡

## 目标

把用户输入快速变成可执行任务，不把 PM 过程变成阻塞开发的流程。

## 任务前必须想清楚

- 目标：这次要交付什么。
- 非目标：明确不做什么。
- 现状：已有项目、文件、文档、接口、设计或限制。
- 影响范围：会碰哪些模块、页面、接口、数据或配置。
- 修改方案：准备怎么做，是否有更简单方案。
- 验收标准：怎么证明完成。
- 风险/回滚：失败或误改时怎么恢复。

## 短任务卡

```markdown
## 任务卡
- 目标：
- 已知事实：
- 不确定/需确认：
- 影响范围：
- 修改方案：
- 验收标准：
- 风险/回滚：
```

## 何时提问

只问会影响执行的关键问题：

- 目标或范围有多种理解。
- 修改可能影响数据、安全、权限、部署、费用、用户体验或不可逆操作。
- 用户给的参考资料用途不清，会影响实现方向。
- 验收标准不清，无法判断完成。

低风险且能合理判断时，直接说明假设并执行。

## 事实分层

当有建议或推断时，用三类表达：

- 已确认：用户明确说过或仓库文档/代码证明。
- 我的建议：Codex 推断或方案，需要用户确认时标明。
- 待确认：会阻塞或显著影响实现的问题。

不要把建议写成用户已确认事实。
