# 技术方案与 ADR

## 目标

技术方案用于把 PRD、产品原型、设计规范和调研结论翻译成可开发、可测试、可回滚的实现契约。它不是为了写大文档，而是提前发现架构、数据、接口、权限、部署和依赖风险。

## 按流程档位执行

### Lean

默认不单独创建 `TECH_SPEC.md`。

只要在 Handoff 中写清：

- 技术栈或运行方式。
- 关键文件范围。
- 验证方式。
- 风险和回滚。

如果涉及数据模型、API、权限、上传、支付、发布或新增依赖，必须升级到 Standard。

### Standard

涉及新项目、核心功能、API、数据结构、权限、状态管理、部署、第三方服务或新增依赖时，必须写一页 `TECH_SPEC.md`。

至少包含：

- 技术栈与选择理由。
- 模块边界。
- 数据模型或数据结构。
- API / 接口契约。
- 状态管理与关键流程。
- 第三方依赖和替代方案。
- 测试与回滚策略。
- 主要风险和待确认项。

### Enterprise

必须写 `TECH_SPEC.md`，并把关键技术取舍沉淀为 ADR 或 `DECISIONS.md`。

必须额外确认：

- 权限模型和数据边界。
- 部署、环境变量、密钥、日志、监控和回滚。
- 数据迁移、兼容和降级方案。
- 多线程/多模块合并顺序。
- 安全、隐私、性能和可扩展性风险。

## ADR 规则

只有影响长期维护的技术决定才写 ADR，不要把普通实现细节都写成 ADR。

需要 ADR 的情况：

- 技术栈选择。
- 数据库、存储、队列、支付、登录、AI 服务等核心依赖。
- API 契约、权限模型、部署方式。
- 放弃某个开源底座或决定二开某个开源项目。

ADR 最小格式：

```markdown
## ADR: [决策标题]

- 状态：proposed / accepted / rejected / superseded
- 背景：
- 选择：
- 替代方案：
- 影响：
- 回滚或退出方案：
```

## 通过条件

- 技术方案能支撑 PRD 和设计规范。
- 修改范围和禁止范围清楚。
- 数据、接口、权限、依赖和部署风险已标出。
- 每个关键风险都有验证或回滚方式。
- Execution Thread 能直接按方案拆 Task。
