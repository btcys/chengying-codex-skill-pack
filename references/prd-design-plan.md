## PRD 完整性门槛

PM Thread 必须逼问出完整、可实现、可验收的 PRD。不能只有一句需求摘要。

Level 2/3 项目在 PRD 冻结前必须先完成竞品、同类产品和开源方案方向校准；PRD 必须吸收调研结论，或明确写明跳过原因和风险。

PRD 至少包含：

- 项目背景：为什么做、解决什么问题、当前替代方案是什么。
- 目标用户：角色、使用频率、设备环境、权限差异。
- 核心场景：用户在什么情况下进入，完成什么任务，成功标志是什么。
- MVP 范围：本版本必须做什么。
- 非目标：本版本明确不做什么。
- 功能清单：按模块列出功能、优先级、输入、输出、状态。
- 用户流程：核心路径、分支路径、异常路径。
- 页面/端清单：Web、App、小程序、后台、移动端、桌面端的范围。
- 数据与权限：对象、字段、角色、权限、可见性、审计要求。
- 集成依赖：第三方登录、支付、地图、消息、AI、存储、推送等。
- 验收标准：每个核心功能如何验证通过。
- 调研影响：竞品/同类产品/开源方案结论如何影响 MVP、非目标、功能优先级、信息架构、设计规范和技术路线。
- 风险与约束：时间、成本、合规、性能、兼容、上线条件。

如果用户回答模糊，必须继续追问。不要用“后面再说”掩盖 PRD 缺口。

## PRD 前期调研输入

调研不是 PRD 后的装饰材料，而是 PRD 形成过程中的判断依据。PM Thread 必须把 `PRODUCT_RESEARCH.md` 或等价调研结论用于回答：

- 竞品和同类产品里哪些能力是用户自然预期。
- 哪些常见能力当前 MVP 不该做。
- 哪些交互、信息架构、筛选、导入导出、权限或状态流转值得参考。
- 是否有成熟组件、模板或可商用开源底座能加速。
- 如果不采用开源方案，原因是 License、架构绑定、维护风险、二开成本，还是收益不足。

## 产品原型完整性门槛

新软件、产品、网站、App、小程序、SaaS、后台、企业内部系统或工具软件，默认必须在正式开发前形成产品原型或交互草图。原型不是只为“UI 好看”，而是提前确认用户怎么完成任务、页面/模块怎么组织、关键状态怎么流转，避免代码阶段才发现产品路径不成立。

产品原型至少回答：

- 用户入口：用户从哪里开始，第一眼看到什么。
- 核心路径：用户完成主要任务的步骤。
- 页面/模块结构：有哪些页面、面板、弹窗、列表、详情、表单或工具区。
- 关键交互：创建、编辑、删除、筛选、上传、导出、支付、分享、协作、审核等动作在哪里发生。
- 状态与异常：loading、empty、error、success、权限不足、数据为空、网络失败、超限、撤销/回滚。
- 信息层级：哪些信息最重要，哪些可以折叠、后置或隐藏。
- 端与适配：移动端、桌面端、响应式、小程序、App 或后台的主要布局差异。

原型深度按流程档位调整：

- Lean：可以用文本流程、页面清单、ASCII/Markdown 草图或 1 到 2 张低保真草图表达；不能只写一句“做个页面”。
- Standard：必须覆盖核心用户路径、关键页面/状态和主要交互；推荐生成低保真原型图、Product Design 方案或 Figma/截图参考。
- Enterprise：必须覆盖角色差异、权限、异常状态、跨模块流转、审计/回滚、移动端或多端适配；必要时拆独立设计/原型确认阶段。

可跳过原型的情况：

- 纯后端 API、CLI、脚本、库、数据迁移、配置修改、测试补齐或低风险 bugfix。
- 已有可信 Figma、截图、旧系统页面、客户原型、设计稿或产品文档，并且已经确认能作为本轮实现依据。
- 本轮明确只做技术验证，不交付用户可操作产品界面。

跳过时必须写明原因和替代依据，例如接口契约、命令行输入输出、现有页面、已有截图或验收清单。不能因为“先写代码再说”跳过。

## 设计规范完整性门槛

有用户可见界面的项目必须逼问出完整设计规范。设计规范建立在产品原型之上；原型确认“做什么、怎么走”，设计规范确认“看起来、用起来是什么标准”。

设计规范至少包含：

- 目标气质：专业、轻量、消费级、工具型、企业级、内容型等。
- 视觉依据：参考产品、截图、Figma、原型、草图、品牌素材、现有页面或竞品 URL；如果没有，必须记录用户确认“无参考，走方案探索”。
- 参考产品：采用什么，不采用什么。
- 信息架构：导航、页面层级、入口优先级。
- 页面清单：每个页面的目标、内容、主要操作。
- 布局规则：栅格、间距、密度、断点、移动端适配。
- 组件规范：按钮、表单、表格、卡片、弹窗、筛选、上传、通知。
- 色彩与主题：主色、中性色、语义色、深浅色策略。
- 字体与层级：标题、正文、辅助文字、数字、空状态文案。
- 交互状态：hover、active、disabled、loading、empty、error、success。
- 动效原则：哪些需要动效，哪些禁止过度动效。
- 可访问性：对比度、键盘操作、焦点状态、文案可读性。

如果用户没有设计偏好，必须先询问是否能补充产品原型、参考截图、Figma、草图、品牌素材或竞品 URL；仍没有参考时，必须给 2 到 3 个方案让用户选择，或调用 Product Design 生成 3 个方向，而不是简单了事。

设计规范通过门槛：

- 已询问用户是否有参考产品、截图、Figma、原型、草图、品牌素材、现有页面或竞品 URL。
- 已确认产品原型或交互草图；若跳过，已写明原因和替代依据。
- 已确认 design brief：产品目标、目标用户、页面/流程、视觉方向、交互强度和端类型。
- 已确认设计依据来源：用户提供 / 竞品调研 / Product Design 方案 / 明确无参考。
- 若需要 UI 视觉目标，已让用户确认是否调用 Product Design；调用后必须先完成 `Product Design:get-context` brief 确认，再用 `Product Design:ideate` 生成 3 个方向并等待选择。
- 已确定选中的视觉方向，或明确当前阶段只做非视觉/非 UI 工作。
- 已把设计验收点写入阶段计划和 QA 验证矩阵。

## 截图与视觉依据存档

用户提供参考截图、修改截图、旧系统截图、设计稿截图、Figma 导出图、竞品截图或验收截图时，PM Thread 必须先判断这些图片是否作为本轮实现或验收依据。

如果作为依据：

- 可访问到本地文件时，复制到 `docs/codex/assets/visual-references/`，建议命名为 `VR-001-<页面>-<用途>.png`。
- 只能看到聊天图片、远程链接或暂时无法复制时，先在 `VISUAL_REFERENCES.md` 记录来源、消息位置/URL、用途、页面/模块和待补文件状态；进入 UI 实现前，必要时要求用户提供可归档文件。
- 截图包含敏感信息时，先脱敏再归档；不能脱敏时只记录来源和不可入库原因，不提交原图。
- 每张图必须在 `VISUAL_REFERENCES.md` 写清：ID、存档路径或来源、页面/模块、对照点、禁止点、确认状态。
- Handoff 中必须引用相关截图 ID 或路径，Execution Thread 才能按图对照实现。

执行或验收阶段产生的 before/after、浏览器 smoke、视觉回归或修复复验截图，归档到 `docs/codex/assets/qa/<Work ID>/`，并在 `VALIDATION.md`、`PHASE_ACCEPTANCE.md` 或 `CODEX_QA.md` 中记录路径。

## 设计工具调用规则

涉及新软件、产品、网站、App、小程序、后台、SaaS、交互原型或视觉方向时，必须判断是否调用设计相关 Skill / Plugin。不要只写“高级感”“简洁”这类空泛描述。

推荐调用顺序：

1. `ui-ux-design-advisor`：用于补齐设计规范、视觉方向、组件规范、布局密度、交互状态和可访问性检查。
2. `Product Design:get-context`：当需要产品原型、UI 设计、视觉探索、重设计或交互页面草图时，先用它确认 design brief。没有明确 brief 时不得进入视觉生成或 UI 实现。PM 必须把 brief 播放给用户确认。
3. `Product Design:ideate`：当没有明确产品原型/视觉目标、用户需要方案选择或需要原型图/页面草图时，在 brief 确认后生成 3 个独立方向，并等待用户选择。
4. `motion-quality`：当页面包含动效、过渡、加载、hover/press、弹窗、抽屉、路由切换、移动端手势或需要“高级感”时，定义动效角色、强度、时长、降级和验证要求。
5. `better-icons`：当界面需要工具栏、导航、状态、操作按钮或图标体系时，用它搜索/选择一致图标库；不要临时拼凑不同风格图标。
6. `imagegen`：用于非 UI 视觉资产、情绪板、插图、封面、素材或 Product Design 不可用时的补充生图；不要用它替代 design brief。
7. `figma:*`：当用户提供 Figma、要求写入 Figma、生成 FigJam / Figma 设计稿、设计系统或图表时使用对应 Figma Skill。
8. `aesthetic-audit` 或 `Product Design:design-qa`：用于 UI 审查、设计还原和视觉质量检查；不能替代 Phase Acceptance 或需求一致性审核。

边界：

- PM Thread 可以调用设计类 Skill 来补齐 brief、方案、原型图和设计规范，但不得写代码。
- `Product Design:image-to-code` 只能在用户选定产品原型/视觉目标、Goal Prompt 获得执行授权、并进入 Execution Thread 后使用。
- 如果 Product Design 插件不可用，PM Thread 必须说明不可用原因，并改为让用户补充参考图/截图/Figma/原型，或用文本方案选择作为降级方案。
- `motion-quality` 在 PM 阶段只产出动效原则和验收项；具体实现和浏览器验证必须在 Execution Thread。
- `better-icons` 在 PM 阶段只确定图标风格和候选库；具体下载、同步或代码接入必须在 Execution Thread。
- 如果用户已有明确设计稿、截图、Figma 或品牌规范，应先读取/确认这些依据，再生成方案。
- 如果项目没有用户可见界面，明确记录“无用户可见界面，跳过设计工具调用”。

## 方案选择机制

当信息不足时，给用户可选方案：

```markdown
这里有 3 个方向可以选：

A. 稳健工具型
- 适合：
- 优点：
- 代价：

B. 商业 SaaS 型
- 适合：
- 优点：
- 代价：

C. 轻量消费型
- 适合：
- 优点：
- 代价：

你选 A/B/C，或告诉我混合方向。
```

技术栈、设计风格、MVP 范围、发布节奏、线程策略都可以用这种方式逼近决策。

## 原型图与交互草图

需要视觉确认时，必须建议生成：

- 首页/核心工作台草图
- 核心流程页面草图
- 移动端关键页面草图
- 后台列表/详情/表单草图
- 关键空状态、错误状态、加载状态

如果当前环境有 Product Design、生图、Figma、截图或设计工具，应按“设计工具调用规则”生成产品原型图或交互页面草图。

新软件/产品/网站没有任何产品原型、参考图、截图、Figma、原型或已选方向时，不得进入产品实现；只能继续 PM 追问、调用 Product Design 生成方向，或明确把原型/界面实现后置。

生成前必须明确：

- 目标页面
- 设备尺寸
- 信息层级
- 风格方向
- 必须出现的模块
- 禁止出现的风格

生成后必须让用户确认，不要把未经确认的视觉方案直接进入开发。

## 阶段开发计划

开发计划必须完整、连续、可验证，不能零散地“这里做做、那里做做”。

每个阶段必须包含：

- 阶段目标
- 输入依赖
- 具体任务
- 修改范围
- 禁止修改范围
- 验收标准
- 自动测试方式
- 人工验收点
- 风险点
- 回滚方式
- 产出文档

阶段顺序建议：

1. 项目基础与架构契约
2. 数据模型与权限基础
3. 核心用户路径
4. 次级功能与边界状态
5. UI 细化与响应式
6. 隐私、安全、性能与兼容检查
7. 打包发布与版本文档

如果项目很小，可以合并阶段，但不能省略验收和回滚。

## Goal Prompt 与执行授权

Goal 是正式开发前的最后确认门禁之一，不应在 PM 早期凭模糊想法提前冻结。这里必须区分三件事：

- Goal Draft：PM 内部或对话中的草案，用来收敛目标和范围，不产生执行授权。
- Goal Prompt：发给用户确认、或让用户复制给 Codex 设置目标的一段话，是交付给用户的授权话术。
- Execution Authorization：用户明确同意“按这个 Goal 执行 / 开始开发 / 进入 Execution”的动作，只有它才能触发执行。

允许先形成 Goal Draft，但最终 Goal Prompt 必须基于：

- 已确认 PRD
- 已确认 PRD 前期竞品/同类产品/开源方案调研结论，或明确跳过原因
- 已确认产品原型/交互草图，或明确说明当前任务无需原型
- 已确认设计规范，或明确说明当前任务没有用户可见界面；涉及界面时，已有参考/原型/截图/Figma/视觉方向，或用户明确确认跳过视觉探索
- 已确认 MVP 与非目标
- 已确认项目等级和 Lean / Standard / Enterprise 流程档位
- 已确认技术方案 / ADR，或明确说明当前任务不需要单独技术方案
- 架构影响面和风险
- 阶段开发计划
- 验证方案和 QA 验证矩阵
- 执行前线程方案：single-thread / N threads / defer 及理由
- 自动化模式
- 开发前对齐包 / Handoff 草案

Goal Prompt 输出必须是一段话，不要写成长文档。它的目标是让用户确认本轮开发授权，或复制给 Codex 设置目标来实施；它不替代 PM Handoff 工作单，也不代表已经开始执行。若用户要把这段 Goal 交给另一个 Codex 执行，必须同时附上或引用开发前对齐包 / Handoff 草案。

模板：

```markdown
Goal Prompt：本轮建议基于已确认的开发前对齐包 / Handoff 草案，按 PRD、产品原型/交互草图、设计规范、技术方案/ADR、架构影响面、阶段开发计划、执行前线程方案和 QA 验证矩阵，完成 [当前版本/当前阶段目标]；只做 [允许范围]，不做 [不做范围]；自动化模式建议为 [Manual / Assisted Autopilot / Long-run Autopilot]。如果你确认“按这个 Goal 执行”，我会登记 Work ID，冻结或补齐 PM Handoff，再进入 Execution Thread；如果你只是想拿这段话去设置目标，请把开发前对齐包 / Handoff 草案一起提供给执行线程。是否按这个 Goal 正式进入开发？
```

用户未明确授权最终 Goal Prompt 前，不得进入 Execution Thread。用户只是发来需求、PRD、目标描述、原型要求或要求“生成 Goal”，都不算确认。不得因为用户没选 Goal 就停止协作；PM Thread 应继续基于阶段开发计划输出：

- 当前阶段状态。
- 还缺哪些信息或确认。
- 推荐下一步。
- 可选方案及影响。
- 是否需要调整 PRD、产品原型、设计规范、技术方案、验证方案或阶段计划。
