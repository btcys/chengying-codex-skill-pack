# 项目开发前确认台

项目开发前确认台是 PM Thread 的临时辅助页面，用来在开发前收集用户对多个确认项的表态。它只解决“对话里逐条回复太乱”的问题，不是项目管理系统，不是文档编辑器，也不是最终 Goal 授权入口。

模板路径：

- `assets/templates/confirmation-board/project-confirmation-board.html`

## 触发条件

出现以下任一情况时，PM Thread 可以提供确认台模板：

- 开发前确认项较多，用户在聊天中逐条回复困难。
- PM 对齐超过两轮，且用户质疑“还缺什么 / 现在到哪一步 / 怎么确认”。
- Codex 给出多个建议，需要用户逐项选择“确认 / 采纳 / 需修改 / 拒绝 / 后置”。
- Standard / Enterprise 项目已经进入最终 Goal Prompt 前的 PRD、设计、技术方案、阶段计划、验收计划或 Handoff 对齐阶段。

Lean 小任务默认不用确认台；Markdown 确认表足够时，不要强行打开页面。

## 使用规则

PM Thread 使用确认台前，必须先把确认项来自哪里说清楚：

- PRD：目标、MVP、非目标、用户路径、验收口径。
- 设计：视觉方向、采用点、不采用点、待改点、是否冻结。
- TECH_SPEC：技术边界、数据源、权限、密钥、部署、外部服务。
- PHASE_PLAN：计划版本、阶段版本、执行顺序、独立审计点、独立验收标准。
- VALIDATION：必跑命令、浏览器截图、Docker、权限、安全、替代验证。
- Handoff：Execution Thread 目标、边界、输入输出、验证要求、回滚方式。

用户在页面完成选择后，把“给 Codex 的回复”复制回对话。Codex 根据回复更新正式文档或对齐包；页面本身不保存到项目正式文档中。

## 硬边界

- 不展示开发进度，不做项目资料进度包，不接入长期项目管理。
- 不允许出现“最终 Goal 授权 / 允许执行 / 开始开发”按钮。
- 不替代 PRD、DESIGN_DECISION、TECH_SPEC、PHASE_PLAN、VALIDATION、Handoff。
- 不要求用户直接编辑 JSON、Markdown 源文件或项目文档。
- 页面不可用、浏览器不可用或用户不想打开页面时，必须降级为 Markdown 确认表继续推进，不能阻塞流程。
- 用户选择“拒绝”时必须填写原因；Codex 后续必须把原因转成文档修改、待确认项或明确的非目标。

## Markdown 降级模板

```markdown
# 开发前确认表

执行授权：未请求。本表只做开发前对齐反馈，不代表允许进入开发。

| 阶段 | 确认项 | Codex 建议 | 用户选择 | 用户说明 |
| --- | --- | --- | --- | --- |
| PRD | MVP 范围 | 只覆盖最小可交付路径 | 待确认 |  |
| 设计 | 视觉方向是否冻结 | 记录采用点、不采用点、待改点 | 待确认 |  |
| 技术方案 | 外部服务与密钥 | 涉及 Key/Token/部署时升级 Standard | 待确认 |  |
| 阶段计划 | 阶段版本与顺序 | 使用 0.0.1 / 0.1.0 / 0.4.5 | 待确认 |  |
| 验收计划 | 阻塞验证项 | 缺截图/Docker/权限验证不得验收通过 | 待确认 |  |
| Handoff | Execution Thread 派发 | Level 2/3 不由 PM Thread 自己改代码 | 待确认 |  |
```
