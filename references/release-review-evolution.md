## 隐私审计

正式交付或对外发布前必须审计：

- 收集哪些用户数据。
- 数据存在哪里，是否加密或脱敏。
- 上传文件是否限制类型、大小、权限和生命周期。
- 日志是否泄露手机号、邮箱、token、API Key、用户内容。
- 第三方 SDK 是否采集额外信息。
- AI 调用是否传入用户敏感数据。
- 权限判断是否在后端强制执行。
- 导出、分享、公开链接是否有访问控制。
- 是否需要隐私政策、用户协议或合规提示。

输出：

- 隐私风险清单
- 必须修复项
- 可后置项
- 发布阻塞项

存在发布阻塞项时，`workflow-gates.md` 的 Release 阶段必须标记为 `blocked`。

## 打包发布

发布前必须确认：

- 版本号
- 发布环境
- 构建命令
- 环境变量
- 数据迁移
- 静态资源与上传路径
- 回滚方式
- 发布后验证路径

发布不是“build 过了就结束”。发布后必须验证核心用户路径。

## 版本文档

每个正式版本必须整理：

- 版本号
- 发布日期
- 目标
- 新增功能
- 修复问题
- 架构变化
- 数据变化
- 隐私与安全审计结果
- 测试结果
- 已知问题
- 回滚方案

推荐写入 `docs/codex/CHANGELOG.md`、`CODEX_QA.md`、`SECURITY.md` 和 `CONTEXT.md`。

## 独立 Code Review Thread

复杂项目或重要版本必须启用独立代码审查线程。

Standard 项目命中以下任一条件时，必须启用 Code Review Thread 或等价独立代码审查：

- 认证、登录、后台账号、角色权限或访问控制。
- API 契约、数据模型、数据库迁移、队列、定时任务或外部数据源。
- 密钥、API Key、Token、支付、上传、公开分享、客户数据或隐私数据。
- 部署发布、服务器配置、构建配置、CI/CD、Docker 或生产环境变更。
- 公共组件、共享模块、跨模块契约、多线程合并或主链路重构。
- 安全、隐私、性能、可用性风险，或用户明确要求审查。

Standard 未命中以上条件时，可以跳过 Code Review，但必须在 Phase Acceptance 或版本记录中写明跳过原因、剩余风险和已运行验证。Enterprise 不允许因“已通过测试”跳过 Code Review。

进入 Code Review Thread 前，Execution Thread 必须已有测试结果和阶段状态，并且需求一致性审核已经通过；没有测试结果或需求一致性审核未通过时不得进入审查。

## 需求一致性审核与 Code Review 的关系

- 需求一致性审核先于 Code Review。
- 需求一致性审核检查“是不是严格按需求做”。
- Code Review 检查“代码是否正确、安全、可维护”。
- 需求不一致的问题不能用 Code Review 通过来掩盖。
- Code Review 发现需求理解偏差时，必须退回需求一致性审核或 PM Thread。

Code Review Thread 职责：

- 审查正确性、安全、性能、可维护性。
- 审查是否越过修改范围。
- 审查架构影响面和上下游依赖。
- 审查测试是否覆盖主链路和边界状态。
- 审查隐私、权限、上传、密钥、日志风险。

Code Review Thread 禁止：

- 直接做大范围业务开发。
- 未经确认直接重构。
- 把审查意见混成新需求。

输出必须按严重程度排序：

- P0：阻塞发布
- P1：高风险，建议本阶段修复
- P2：可后置但要记录

存在 P0 时不得进入 Release Thread。

Code Review 打回必须输出 Fix Request：

```markdown
## Code Review Fix Request

| ID | 严重级别 | 类别 | 问题 | 证据 | 期望结果 | 必须验证 |
| --- | --- | --- | --- | --- | --- | --- |
| FX-CR-1 | P0/P1/P2 | correctness/security/performance/maintainability/test |  |  |  |  |
```

- P0 必须回 Execution Thread 修复，不得后置。
- P1 默认本阶段修复；如后置必须由用户或 PM Thread 明确确认。
- P2 可后置，但必须写入 `REVIEW_CHECKLIST.md`、`TASKS.md` 或版本文档。
- 同一 Code Review 问题最多自动修复 3 轮；仍不通过时回 PM Thread 或技术方案重新确认。

## 版本关闭

Release Thread 完成后，当前版本应进入关闭状态。

版本关闭必须记录：

- 版本目标是否完成。
- 需求一致性审核是否通过。
- Code Review 是否通过。
- 隐私审计是否通过。
- 发布后主链路是否通过。
- 是否存在后置项。
- 是否需要新 PM Thread。

当前版本关闭后，不自动回 PM。只有出现新需求、范围变更、返工、下一版本规划或用户明确要求，才重新进入 PM Thread。

## 用户反馈驱动自进化

自进化不是版本结束后的泛泛复盘，而是用户反馈触发的持续修正规则。用户指出流程问题、执行偏差、遗漏、重复错误或协作方式问题时，必须立即判断是否需要记录。

触发条件：

- 用户明确反馈“这里不合理、没按要求、又犯了、以后要记住”。
- 同类问题第二次出现。
- 一次问题导致返工、验收失败、需求偏离或发布风险。
- Code Review、Phase Acceptance、Requirement Compliance 发现重复缺口。
- 用户纠正 Codex 的协作方式，例如阿谀奉承、跳过风险、未经确认推进。

处理流程：

1. 记录原始反馈到 `docs/codex/FEEDBACK_LOG.md`，包含日期、阶段、问题、影响、来源和当前处理状态。
2. 判断反馈类型：需求规则、设计规则、架构规则、测试规则、协作规则、发布规则或安全隐私规则。
3. 判断升级等级：
   - 一次普通反馈：仅记录到 `FEEDBACK_LOG.md`。
   - 第二次同类反馈，或一次造成返工：升级到 `REVIEW_CHECKLIST.md` 或相关专题文档。
   - 第三次同类反馈，或影响全项目流程：升级到 `AGENTS.md` 或流程门禁。
4. 更新对应文档，而不是只在聊天里承诺。
5. 在下一次同类任务开始前，先检查 `FEEDBACK_LOG.md` 和 `REVIEW_CHECKLIST.md` 是否有相关规则。

文档归属：

- `docs/codex/FEEDBACK_LOG.md`：原始反馈、重复次数、处理状态和升级去向。
- `docs/codex/REVIEW_CHECKLIST.md`：以后每次都要检查的问题。
- `docs/codex/DECISIONS.md`：已经确认的产品、技术或流程决策。
- `docs/codex/CONTEXT.md`：长期上下文、用户偏好、项目特殊约束。
- `docs/codex/SECURITY.md` / `PRIVACY_AUDIT.md`：安全、隐私、合规类反馈。
- `AGENTS.md`：全项目硬规则；只记录高影响、长期有效、已经确认的规则。

记录标准：

- 只记录可复用、可执行、可检查的规则。
- 不记录情绪化流水账。
- 不把一次性偏好直接升级为全项目硬规则。
- 必须标明来源、适用范围、触发条件、验证方式和升级目标。
- 如果问题来自迎合式确认、未指出风险、跳过反例或无依据乐观承诺，必须记录为协作质量问题，并写入 `REVIEW_CHECKLIST.md`。

反馈记录示例：

```markdown
### 2026-06-18：阶段验收不能由执行线程自验

- 来源：用户反馈
- 阶段：工作流设计
- 问题：Execution Thread 完成后缺少独立验收线程，容易自我通过。
- 影响：可能出现计划未完成仍进入下一阶段。
- 重复次数：2
- 处理：升级到 PHASE_ACCEPTANCE.md 与 AGENTS.md。
- 后续检查：每阶段完成后必须进入 Phase Acceptance Thread。
```

协作质量记录示例：

```markdown
### 2026-06-18：禁止阿谀奉承式协作

- 来源：用户反馈
- 阶段：全流程协作
- 问题：不能用户说什么都顺着说，必须指出风险、矛盾和不可实现点。
- 影响：会弱化方案质量和验收边界。
- 重复次数：2
- 处理：升级到 AGENTS.md 与 REVIEW_CHECKLIST.md。
- 后续检查：PM、方案评估、代码审查、发布决策前检查是否有迎合式确认。
```
