# 验证策略

验证用于证明任务完成。不要因为验证文档拖慢开发，但也不能把未验证写成已通过。

## 默认验证顺序

1. 读取项目已有脚本：`test`、`lint`、`typecheck`、`build`、`e2e`、`dev`。
2. 运行和本次修改最相关的最小命令。
3. UI/前端/可视化改动做浏览器检查或截图。
4. 涉及权限/API/数据/部署/安全时补对应 smoke 或人工验证。
5. 没有自动测试时，写替代验证清单。

## UI 截图

优先使用 Codex 内置浏览器、项目 dev server / preview、`localhost` 静态服务或项目已有 e2e 截图工具。缺浏览器、Docker、账号或权限时，先请求必要审批；无法完成时记录已尝试路径和风险。

## 输出格式

```markdown
## 验证结果
- 已运行：
- 结果：
- 未验证：
- 原因：
- 风险：
- 下一步：
```

验证失败时继续修复；无法修复时说明阻塞。
