# THREADS.md

## 多线程开发状态

> Level 2 / Level 3 默认启用 PM 与 Execution 分离。
> 长期多线程治理主要用于 Level 3 复杂项目；临时线程必须在任务结束后归档。
> PM Thread 与执行线程必须分离。

## PM Thread

状态：默认启用  
职责：需求梳理、范围确认、项目分级、任务拆解、风险说明、验收标准、交接  
规则：不允许改代码；Level 2 / Level 3 必须在用户确认进入开发后安排 Execution Thread 或已有执行线程

### 交接包

- 项目一句话说明
- 项目等级
- MVP 与非目标
- 当前任务目标
- 修改范围
- 禁止修改范围
- 验收标准
- 风险与回滚提示

## 长期线程

### Long-lived Core Thread

状态：未启用  
职责：核心架构、状态管理、数据模型、领域服务  
风险：高  
合线要求：必须双重检查

### Long-lived Feature Threads

状态：未启用  
职责：稳定功能模块开发，适合跨阶段持续迭代  
规则：每个线程必须绑定明确修改范围，并说明其长期负责的模块

## 临时线程

### Temporary Hotfix Thread

状态：未启用  
职责：紧急修复  
规则：必须可快速回滚，修复完成后立即合并归档

### Temporary Experiment Thread

状态：未启用  
职责：实验性方案  
规则：默认不长期保留，结论确认后必须合并或放弃，并归档记录结论

### Temporary Spike Thread

状态：未启用  
职责：技术调研、方案验证、开源验证  
规则：验证结论沉淀后关闭，不应长期悬挂

## 归档规则

- 临时线程必须标注开始时间、目标、预计收口条件
- 临时线程完成后必须标注结果：已合并 / 已放弃 / 已归档
- 合并后需要在 MERGE_PLAN.md 记录合线顺序与回滚点
- 已结束的临时线程不应继续承担新任务，应新开线程或并入长期线程
